A function is created to handle specific string arrays in data
```python
def parse_features(x):
    if x == "{}":
        return []

    # Remove { and }
    x = x.strip("{}")

    # Split on comma outside quotes
    items = re.findall(r'"(.*?)"', x)

    # Fix escaped characters
    items = [i.replace("\\'", "'").replace("\\\\", "\\") for i in items]

    return items
```
The data is processed efficiently by reading the large CSV file in small chunks to avoid memory overload, while only selecting the necessary columns and skipping corrupted rows that break CSV formatting rules. Missing values are handled by replacing them with 'Unknown', and additional cleaning is applied to specific fields such as features. The cleaned data is then written incrementally to the output file to ensure smooth processing and prevent duplication of headers.

```python
def load_clean(output_path="song_lyrics_clean.csv"):
    if os.path.exists(output_path):
        return

    for i, chunk in enumerate(pd.read_csv(
        "song_lyrics.csv",
        engine="python",
        usecols=cols_without_lyrics,
        on_bad_lines="skip",
        chunksize=25_000
    )):
        chunk['title'] = chunk['title'].fillna('Unknown')
        chunk['language_cld3'] = chunk['language_cld3'].fillna('Unknown')
        chunk['language_ft'] = chunk['language_ft'].fillna('Unknown')
        chunk['language'] = chunk['language'].fillna('Unknown')
        chunk["features"] = chunk["features"].apply(parse_features)

        chunk.to_csv(output_path, mode="a", header=(i == 0), index=False)

    return

load_clean()
```

## 🛠️ Task 3: Apply Big Data Handling Strategies

### 📊 Performance Measurement Setup

A `summary` dictionary and a `measure_performance` function were implemented to track and log key metrics across all strategies.

**Code**:
```python
summary = {
    'Strategy': [],
    'Row Count': [],
    'Column Count': [],
    'Load Time (s)': [],
    'Peak Memory (MB)': [],
    'Memory Usage (MB)': [],
}

def get_memory_usage(df):
    if isinstance(df, pd.DataFrame):
        return df.memory_usage(deep=True).sum()
    elif isinstance(df, pl.DataFrame):
        return df.estimated_size()
    elif isinstance(df, pyarrow.Table):
        return df.nbytes

def measure_performance(read_func, strategy_name):
    tracemalloc.start()
    start = time.perf_counter()

    df = read_func()

    load_time = time.perf_counter() - start
    current, peak = tracemalloc.get_traced_memory()
    tracemalloc.stop()

    mem_usage = get_memory_usage(df)

    print(f"Strategy: {strategy_name}")
    print(f"Row Count: {df.shape[0]}")
    print(f"Column Count: {df.shape[1]}")
    print(f"Load Time: {load_time:.4f}s")
    print(f"Peak Memory: {peak / 1024 / 1024:.2f} MB")
    print(f"Memory Usage: {mem_usage / 1024**2:.2f} MB")

    summary['Strategy'].append(strategy_name)
    summary['Row Count'].append(df.shape[0])
    summary['Column Count'].append(df.shape[1])
    summary['Load Time (s)'].append(round(load_time, 4))
    summary['Peak Memory (MB)'].append(round(peak / 1024**2, 2))
    summary['Memory Usage (MB)'].append(round(mem_usage / 1024**2, 2))
```

**Explanation**:  
To evaluate the effectiveness of different big data handling strategies, a custom `measure_performance` function was implemented. This function captures key performance metrics such as memory usage (peak and actual DataFrame memory) and load time. A `get_memory_usage` helper handles different DataFrame types for objective comparison between various optimization techniques.

**Implementation Summary**:

* Uses `tracemalloc` to track peak memory allocations during the function call.
* Measures execution time using `time.perf_counter()` for high precision.
* Computes actual in-memory DataFrame size using `memory_usage(deep=True)` or library-specific equivalents.
* Logs and appends all metrics to a shared `summary` dictionary for later comparison.

### 2. Chunking

**Code**:
```python
def chunking():
    chunks = []

    for chunk in pd.read_csv('song_lyrics_clean.csv', chunksize=10000):
      chunk.columns = chunk.columns.str.strip()
      chunks.append(chunk)

    return pd.concat(chunks, ignore_index=True)

measure_performance(chunking, 'Chunks of 10000')
```

**Explanation**:  
This approach loaded the data in smaller parts (chunks) instead of all at once to prevent memory overload and allow processing of datasets that may not fit entirely into memory. Then, combines them together.   

**Implementation Summary**:  
The dataset was loaded in chunks of 10,000 rows using `pandas.read_csv()` with the `chunksize` parameter. Each chunk was processed and then concatenated into a single DataFrame using `pd.concat()`. Column names were also stripped of leading/trailing whitespace for consistency. Chunking is expected to reduce peak RAM usage by limiting the amount of data held in memory at any given time.

#### Library: Dask

**Code**:
```python
def read_dask():
    df = dd.read_csv(
        "song_lyrics_clean.csv",
        engine="python",
        on_bad_lines="skip",
        blocksize="64MB",
        dtype=str
    )

    df = df.compute()

    df['year'] = pd.to_numeric(df['year'], errors='coerce')
    df['views'] = pd.to_numeric(df['views'], errors='coerce')
    df['id'] = pd.to_numeric(df['id'], errors='coerce')

    return df

measure_performance(read_dask, 'Dask')
```

**Explanation**:  
Dask enables parallel processing by breaking large datasets into smaller chunks and processing them concurrently across multiple CPU cores. `dd.read_csv` reads the file lazily in partitions. The `on_bad_lines='skip'` and `engine='python'` options ensure robustness against malformed rows, though the Python engine is slower than the default C engine. The `.compute()` method triggers the actual execution, combining all partitions into a standard pandas DataFrame. Numeric columns are then cast after loading since `dtype=str` was used for uniformity.

## 🧠 Task 5: Conclusion & Reflection

### 🔹 Summary of Observations

Overall, among the five strategies tested on the Genius Song Lyrics dataset (9.07 GB), there are clear trade-offs between performance, memory usage, and data integrity.   

**Sampling** is the fastest and least memory-intensive method, as loading only about 10% of the approximately 5.1 million rows significantly reduces loading time and memory consumption. However, it sacrifices dataset integrity and is therefore unsuitable for comprehensive analyses where all records are critical. 

**Loading less data** offers the best practical balance by removing identifier columns (title, id) and redundant language columns (language_cld3, language_ft), memory and loading time can be reduced without affecting the number of rows available for analysis. Therefore, this is the most recommended strategy when the complete dataset is required but not all columns are relevant.   

**Chunking** is primarily used for memory peak management to process the dataset in batches of 10,000 rows prevents any single memory allocation from exceeding RAM, which is crucial when saving the cleaned CSV file. However, this approach does not improve overall runtime, it increases overhead due to repeated I/O and join operations, making it slower than direct loading.   

**Optimising data types** significantly reduces the in-memory DataFrame size by using compact types like Int16, category, and string, but comes at the cost of noticeably longer parsing time due to strict type validation during ingestion, particularly for nullable integer types. This highlights that not all optimisations improve all metrics simultaneously.   

Inconclusion, in terms of scalability, **selective loading** and **chunking** are the most suitable for handling growing datasets as they better manage memory and maintain stable performance when data size increases without requiring the full dataset to reside in memory at once.

For the library comparison, the four libraries differ mainly in performance, memory efficiency, and tolerance for messy data.    

**Pandas** is the most straightforward and flexible. It handled the raw features column without issue and required no pre-cleaning but struggles with the 9.07 GB dataset due to high memory usage and single-threaded parsing. 

**Polars** and **PyArrow** both delivered the fastest load times and lowest memory usage, owing to their Rust-based and Arrow-native implementations respectively, which leverage multi-threaded columnar reading. However, both are strict CSV parsers that failed on the raw file's malformed features and lyrics columns, requiring the data to be pre-cleaned before benchmarking.    

**Dask** is designed for distributed and out-of-core processing, making it the most scalable option for datasets that exceed available RAM, but for a single local CSV it introduces task-scheduling overhead that makes it the slowest of the four, especially when forced to use the Python engine to handle bad lines.   

Overall, **Pandas** is ideal for smaller or messier datasets where flexibility matters, **Polars** and **PyArrow** are the best choice when speed and memory efficiency are the priority and the data is clean, and **Dask** is most appropriate when the dataset is too large to fit in memory or when distributed processing across a cluster is required.
